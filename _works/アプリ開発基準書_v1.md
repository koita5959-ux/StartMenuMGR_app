# アプリ開発基準書 v1.0

**適用範囲：インストーラー付きで配布するすべてのアプリケーション**
**策定：2026年4月9日**

> **この基準書の位置づけ**
> 本書は全アプリ共通のルールである。特定アプリ固有の仕様・設計・技術選定は、
> 各プロジェクトの「制作指示書」に記載し、本書には含めない。
> 本書を改訂する際は、特定アプリの都合に引っ張られていないか確認すること。

---

## 1. 開発の思想

### 1-1. 「あ、便利…」と思えるツールの開発

テーマは、Vectorや窓の杜で公開されているものと被っても問題ない。
要は「これを使うと無駄なPC作業が効率的になる」と訴求できればよい。

### 1-2. 趣味でぶったツールでなく「アプリ」であること

ネット上で拾ってきた無料ツールの多くは、開発者の思い込みで作られている。
ヒントはまなべど便利でも、多くのツールに言えることは、使ってくれる相手への意識が足りていない。
本業を脇でこなす人たちが作るアプリとして、使ってくれる相手を意識したリリースが重要である。

### 1-3. ソースコードを起点とした開発

多岐に渡るアプリ群の基本はソースコードである。
ソースがないと「直し」も行きに詰まり「作り直す」ことになる。
ソースコードからコンパイル（ビルド）し、インストーラー付きアプリとして完成させる。

## 2. プロジェクト構成

### 2-1. フォルダ構成

標準的で簡素な構成を維持する。

```
AppName_app/                  ← プロジェクトルート
├── AppName.csproj            ← プロジェクトファイル
├── App.xaml / App.xaml.cs    ← アプリエントリポイント
├── MainWindow.xaml / .cs     ← メインウィンドウ
├── Models/                   ← データモデル（必要に応じて）
├── Services/                 ← ビジネスロジック（必要に応じて）
├── ViewModels/               ← ViewModel（必要に応じて）
├── Assets/                   ← アイコン・画像等のリソース
├── _works/                   ← ソースコードに関係しないファイル置き場
│   ├── Trash/                ← 対応済・不要ファイルの退避先
│   ├── 制作指示書等          ← 生きているドキュメント
│   └── ...
└── Installer/                ← Inno Setupスクリプトと出力先
```

### 2-2. `_works` フォルダの運用

- ソースコードに関係しないファイル（制作指示書、メモ、参考資料等）を置く
- `_works` 直下は「生きているデータ」のみとし、常に整理された状態を保つ
- 対応済みや不要になったファイルは `_works/Trash/` に移動する
- `Trash/` は削除ではなく保管。確認が必要になったときに参照できる状態にする

## 3. バージョン管理

### 3-1. バージョン番号

| 形式 | 例 | 説明 |
|------|------|------|
| vメジャー.マイナー | v01.00 | 初版リリース |
| | v01.01 | バグ修正・小さな改善 |
| | v02.00 | 大きな機能追加・設計変更 |

- 初版は `v01.00` から開始する
- 必要に応じてバージョンを上げる（義務的な定期更新はしない）
- バージョン番号はアプリのタイトルバー、About画面、インストーラーに反映する

### 3-2. Git

- ソースコード管理にGitを使用する
- **コミットのタイミング：大きな変更の区切りでユーザーが指示する**
- 細かい自動コミットは行わない。意味のある単位でコミットする

### 3-3. GitHub連携

- GitHub活用時は、コンパイル～インストーラー作成のタイミングで随時連携を実行する
- リリース物のバージョン管理と配布に活用する

## 4. リリース基準

### 4-1. インストーラー

Inno Setup Compilerを使用する。
コンパイル済みのexeをインストーラーに組み、利用者のPCに届ける形にする。
ビルドからインストーラー作成までは、ターミナルから一括実行する。

| 項目 | 内容 |
|------|------|
| インストーラー命名 | `{アプリ名}_setup_v{バージョン}.exe`（例: `StartMenuMGR_setup_v01.01.exe`）|
| スタートメニュー | アプリ名で登録（利用者が見つけやすいこと） |
| デスクトップ | アイコン配置は利用者の選択制 |
| アイコン | カスタムアイコンをexeに組み込み（外部ファイル不要） |

### 4-2. アンインストーラー

インストーラーと同じく重要。**痕跡を残さない**が必須。

| 項目 | 基準 |
|------|------|
| プロセス停止 | 実行中のアプリを安全に停止させる |
| ファイル削除 | インストールフォルダとデータフォルダの両方を削除 |
| レジストリ | アプリが書き込んだレジストリキーがあれば削除 |
| スタートアップ | 自動起動設定があれば解除 |
| 完了確認 | インストール前と同じ状態であることを検証 |

### 4-3. API・OS機能の追加と削除

API（Token）やOS機能・プロセスの追加はインストール時に組み込むことがある。
ただし削除もセットと言える。単純に削除すると他の機能に影響が出る場合があるため、
この点の検証は丁寧に行う必要がある。

## 5. 配布物の構成

ZIP形式で以下を同梱する。
配布ZIPの作成は、コンパイル～インストーラー作成と同時に行う。
同時進行にすることで、抜け漏れを防ぐ。

```
AppName_v01.00.zip
├── AppName_setup_v01.00.exe   ← インストーラー
├── ご利用にあたって.txt        ← ツール説明・制限事項・アンインストール方法
└── テスト実績.txt              ← 対応環境・確認済み動作の記録
```

### 5-1. 「ご利用にあたって.txt」の記載事項

| 項目 | 内容 |
|------|------|
| ツール概要 | 何ができるアプリか（簡潔に） |
| 動作環境 | 対応OS、必要な.NETバージョン等 |
| インストール方法 | Setup.exeの実行手順 |
| アンインストール方法 | 「プログラムの追加と削除」からの手順 |
| 制限事項・既知の問題 | あれば記載 |
| バージョン履歴 | バージョン番号と変更内容（最終確定時に整理） |

### 5-2. 「テスト実績.txt」の記載事項

モニター利用者への配布時に、何が確認済みかを伝えるための記録。

| 項目 | 内容 |
|------|------|
| テスト環境 | OS、マシンスペック等 |
| 確認済み動作 | 動作確認した機能の一覧 |
| 未確認・制限事項 | まだ検証できていない環境や操作 |
| 既知の不具合 | 把握しているが未修正の問題 |

## 6. ClaudeCodeとの協業

### 6-1. 制作指示書

- 構想立案（Claude.AI）が制作指示書を書き、ClaudeCodeが実装する
- 制作指示書は「こう作れ」ではなく「こういうものを作りたい」を伝える
- 技術的な実現手段の選択はClaudeCodeの判断に委ねる
- 指示書は `_works/` に保管する

### 6-2. 相談と判断

- 不明点や確認が必要な場合は、ユーザーに直接相談する
- ユーザーはVSCodeで直接ファイルに触れる共同制作者である
- 構想立案（Claude.AI）への差し戻しが必要な場合は、ユーザーに伝える

### 6-3. トラブル発生時

- **作業を止める。** 修正コードを書かない、言い訳もしない
- 何が起きたか、何が壊れたか、直近の変更箇所はどこか、を事実だけ簡潔に報告する
- 原因の可能性が高いものを**1つだけ**挙げる（確信がなければ「不明」と書く）
- 最小限の修正案を1つ提示し、**ユーザーの承認を得てから**実行する
- 壊れた箇所以外のコードを触らない。「ついでに直す」はしない
- セッション残量が厳しい場合は、正常なコードの状態を保全し、引継ぎメモを出力する

## 7. ドキュメント体系

本開発で使用するドキュメントは以下の2種類のみとする。
ドキュメントを増やすこと自体が負担になるため、これ以上増やさない。

| ドキュメント | 役割 | 置き場所 |
|-------------|------|---------|
| アプリ開発基準書（本書） | 全アプリ共通のルール | 共通の保管場所 + 各プロジェクトの `_works/` |
| 制作指示書 | アプリ固有の仕様・設計・技術選定 | 各プロジェクトの `_works/` |

- 基準書は共通の保管場所を正とし、各プロジェクトにはコピーを置く
- 改訂時は共通の保管場所を更新し、進行中のプロジェクトへ反映する
