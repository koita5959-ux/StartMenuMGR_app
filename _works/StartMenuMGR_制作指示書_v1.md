## 2. 開発背景と動機

### なぜこのツールが必要か

1. **Windows 10/11のスタートメニュー管理の後退**
   - Windows 10のスタートメニューは、ショートカットの配置やフォルダ構成をUIから自由に管理する手段が乏しい
   - A〜Zのアルファベット順に並ぶだけで、ユーザーが構造を自分の意図通りに整理しにくい

2. **自作アプリ開発者としての実需**
   - DesktopKitをはじめとする自作アプリ群が増えている
   - アプリを作るたびに、ショートカット作成→プロパティ設定→スタートメニュー配置を手動で行っている
   - .exe右クリックで「スタートメニューにピン留め」が出るものと出ないものがあり、結局エイリアス経由の手動作業になる
   - AI時代にコード生成で実行型ファイルが増えるスピードに対して、スタートメニュー管理が追いついていない

3. **RPAシリーズの製品化パイプラインの一部**
   - 自作アプリを作る→スタートメニューに整理配置する、というフローの後半を担うツール
   - 開発者自身が最初のヘビーユーザー

### OS対応方針

- **本版はWindows 10専用として設計する**
- Windows 11はスタートメニューの構造がWindows 10と大きく異なる（ピン留め/おすすめ方式、タイルグループ廃止等）
- Windows 11版は将来的に別プロダクトとして開発する可能性がある
- UIの思想や操作感は統一するが、技術的な実装はOS構造に合わせてそれぞれ作る
- Windows 10は2026年中は実用環境として現役（Chromeサポート継続を根拠とする）

## 3. 基本コンセプト：疑似スタートメニュー方式

### 方式の概要

StartMenuMGRは、Windowsのスタートメニュー自体に割り込むのではなく、**アプリ独自のウィンドウ内にスタートメニューの構造を再現（疑似スタートメニュー）し、その中で操作を行い、「適用」で実際のスタートメニューに反映する**方式を採る。

### この方式を採る理由

- Windowsのスタートメニューへの直接的なUI操作介入（ドラッグ拡張、右クリックメニュー追加等）は、システムレベルのフックが必要であり、OSアップデートで動作しなくなるリスクが高い
- Start11やOpen-Shell等の市販ツールはそのレベルの開発を行っているが、専門チームの継続メンテナンスが前提
- 疑似スタートメニュー方式であれば、ファイルシステム上のスタートメニューフォルダの読み書きで実現でき、技術的に安全かつ現実的

### 操作フロー

```
起動 → 現在のスタートメニューの構造を読み取り、疑似スタートメニューに再現
  ↓
ユーザーが疑似スタートメニュー上で操作（移動、フォルダ作成、リネーム、削除、追加等）
  ↓
「適用」ボタンで実際のスタートメニューに反映
  ↓
結果を確認し、必要なら「元に戻す」で起動時の状態に復元
```

## 4. 幹の設計（最優先で確立すべき基盤）

以下の4点がStartMenuMGRの技術基盤であり、最初に確立すべき「幹」である。
個々の操作機能（枝）は、幹が動いてから段階的に追加する。

### 4-1. 疑似スタートメニューUI

- 起動時に、Windows 10のスタートメニューフォルダの構造を読み取る
- 読み取り対象：
  - `C:\ProgramData\Microsoft\Windows\Start Menu\Programs`（全ユーザー共通）
  - `%APPDATA%\Microsoft\Windows\Start Menu\Programs`（現在のユーザー）
- フォルダ構造、ショートカット（.lnk）、その他のファイルを一覧として表示
- 表示はWindows 10のスタートメニュー左側（A〜Zリスト＋フォルダ構造）に近いイメージだが、完全再現は目標としない
- **「操作に必要な構造を視認できるレベル」の再現で十分**
- 見た目の作り込みは、動くものを触ってから調整する

### 4-2. 適用（反映）

- ユーザーの操作結果を、実際のスタートメニューフォルダに反映する
- ファイルシステム上のショートカット移動、フォルダ作成、リネーム、削除として実行
- 管理者権限が必要な操作（全ユーザー共通フォルダへの書き込み等）がありうることに注意

### 4-3. 元に戻す（起動時スナップショット）

- アプリ起動時に、スタートメニューフォルダの状態をスナップショットとして保存する
- 「元に戻す」ボタンで、起動時の状態に復元する
- 適用を何回繰り返しても、戻る先は常に「起動時の状態」
- アプリを閉じた時点で確定。次回起動時に新しい起点を取る
- 複雑なUndo履歴の管理は不要

### 4-4. テーマ対応（最低限）

- Windowsのダークモード/ライトモード設定を自動取得し、UIに反映する
- アクセントカラーの取得・反映ができればなお良い
- フォルダやショートカットのアイコンは実際のものを表示する
- 完全なスタートメニューの見た目再現は目標としない
- 「違和感がない程度」が基準

## 5. 枝の機能（段階的に実装）

以下は幹が確立した後、順次追加していく機能群。
優先度や実装順はClaudeCodeとの相談で決めてよい。

| 機能 | 概要 |
|------|------|
| 移動 | ドラッグ操作でショートカットやフォルダの位置を変更 |
| フォルダ作成 | 右クリックメニューから新規フォルダを作成 |
| リネーム | ショートカットやフォルダの名前を変更 |
| 削除 | ショートカットやフォルダを削除 |
| ショートカット新規登録 | .exe/.bat/.ps1等を選択し、ショートカットを自動生成してスタートメニューに配置 |
| プロパティ編集 | ショートカットのパス、アイコン、説明欄等の編集 |
| グルーピング | フォルダ単位でのアプリのまとめ（#位置への配置含む） |

## 6. 技術的な前提と注意事項

### スタートメニューの実体

Windows 10のスタートメニュー左側（A〜Zリスト）は、以下のフォルダのショートカット（.lnk）で構成されている：

- `C:\ProgramData\Microsoft\Windows\Start Menu\Programs`（全ユーザー共通）
- `%APPDATA%\Microsoft\Windows\Start Menu\Programs`（現在のユーザー）

この2つのフォルダの内容がマージされてスタートメニューに表示される。
フォルダ操作（作成、移動、リネーム、削除）は通常のファイルシステム操作で可能。

### 右側タイルグループについて

Windows 10のスタートメニュー右側のタイルグループ（ライブタイル）は、ファイルシステムではなくレジストリやAPI経由で管理されている。
本版では右側タイルの操作を必須機能としない。左側のA〜Zリスト＋フォルダ構造の管理を優先する。
タイル操作は将来の拡張として検討する。

### 管理者権限

全ユーザー共通フォルダ（ProgramData配下）への書き込みには管理者権限が必要になる場合がある。
アプリの権限設計（起動時にUAC昇格するか、必要時に昇格するか等）はClaudeCodeの判断に委ねる。

### 構想立案の技術的見解の扱い

上記の技術的記述は、構想立案（Claude.AI）の知識に基づくものである。
実際のWindows 10のスタートメニューの動作と異なる部分がある可能性がある。
ClaudeCodeは実装にあたり、実際の動作を検証した上で判断してほしい。
「仕様書にはこう書いてあるが実際は違った」は歓迎するフィードバックである。
